# Spec for PAR-740: Delegate Payment Selection To Checkout Form

branch: feature/PAR-740-Delegate-Payment-Selection-To-Checkout-Form

## Summary

Some merchants using the Gutenberg blocks checkout experience a broken seQura payment flow: the solicitation is never triggered by React, leaving the block content empty and preventing shoppers from selecting a payment method or completing the order. This is a rendering timing issue specific to the blocks-based checkout.

The seQura API supports a mode where payment option selection is delegated to the hosted checkout form itself, activated by passing `tbs` as the product identifier in the checkout form URL. When this mode is active, the payment method selection step on the merchant's page is skipped entirely — the shopper is redirected to the seQura form which handles selection internally. This resolves the empty block problem for affected merchants.

Because delegating selection to the seQura form can reduce conversion (shoppers leave the merchant's page earlier), this behavior must not be the default. It must be opt-in, activated through a WordPress filter that merchants or developers can enable per-site.

## Functional Requirements

- By default, the seQura blocks-based checkout behavior is unchanged: solicitation and payment option selection happen on the merchant's checkout page.
- A WordPress filter must be available that, when applied with a truthy return value, enables the delegated-selection mode for the seQura payment block.
- When delegated-selection mode is enabled, the seQura checkout form URL must include `tbs` as the product parameter, causing the seQura API to handle payment option selection within its own hosted form.
- The filter must be documented (filter name, expected return type, and its effect) so merchant developers can enable it without needing to read source code.
- The filter must be evaluated at checkout time, not at configuration time, so it can be toggled without clearing caches or changing plugin settings.
- The feature must not require any new admin UI, settings field, or database entry.

## Possible Edge Cases

- A merchant enables the filter on a classic (non-blocks) checkout: delegated-selection mode should either be ignored or behave consistently without breaking the classic flow.
- The filter is conditionally applied (e.g., only for certain product categories or user roles): the URL parameter must reflect the filter result at the time of each individual checkout session.
- The `tbs` product parameter conflicts with or overrides other product parameters already set in the checkout form URL construction logic.
- Merchants using both blocks and classic checkout simultaneously (e.g., different pages) expect the filter to apply globally or per-page — behavior must be predictable and documented.
- The solicitation request payload may also need to reflect the delegated mode, not just the redirect URL.

## Acceptance Criteria

- With the filter absent or returning falsy, the blocks checkout behaves exactly as before — no regression.
- With the filter returning truthy, the seQura checkout form URL contains `tbs` as the product, and shoppers are redirected to the seQura-hosted selection form.
- Enabling the filter on a classic checkout does not cause errors or unexpected behavior.
- The filter name follows WordPress naming conventions and is prefixed with `sequra_` to avoid conflicts.
- No new database records, admin settings, or REST endpoints are introduced.

## Open Questions

- Should the filter also affect the classic (non-blocks) checkout, or be blocks-only? This matters for merchants using both modes.
- Does passing `tbs` as the product require any change to the solicitation request body (not just the redirect URL), or is the URL change alone sufficient?
- Should the filter receive any arguments (e.g., WC order object, payment method ID) to allow conditional logic in the filter callback?
- Is there a defined filter naming convention already in use in this plugin that should be followed?

1. **Scope**: Both classic and blocks checkout. Classic renders a single hidden input with `product=tbs` when delegated mode is active.
2. **Filter**: `sequra_delegate_payment_method_selection(bool $enabled, ?\WC_Order $order)`, following the newer `sequra_*` naming convention.
3. **Filter evaluation**: Per-session on every checkout interaction. In blocks, evaluated in PHP at script-localization time and passed as a boolean (`isDelegatedSelection`) in `SeQuraBlockIntegration` so the JS block can bypass AJAX without any server round-trip. In classic, evaluated at `payment_fields()` render time, at `validate_fields()`, and at `process_payment()`.
4. **UI text**: No sub-options when delegated — just the existing gateway title "seQura" without any radio buttons or method list.
5. **Helper webhook**: `Toggle_Delegated_Selection_Task` added to `sequra-helper/` using a `wp_options` flag and a filter hook, matching the pattern of `Toggle_Cache_Task`.
6. **Playwright**: New spec `007-checkout-delegated-selection.spec.mjs` verifying: filter-off = existing multi-method flow unchanged; filter-on = form URL contains `product=tbs` and order completes.
7. **Product value**: Inline string literal `'tbs'` — no constant defined, as the value may be removed or treated as default by the API in future.
8. **Integration-core**: `GetFormRequest` accepts `?string $product` — confirmed that passing `'tbs'` works with the locked `v4.1.0` vendor version with no vendor modifications needed. The `GetFormHttpRequest::transformQueryParameters()` includes `product` in the URL only when truthy, so `'tbs'` will appear as `?product=tbs`.

### Risks & Concerns

_(2026-04-20: logo-display risks added)_
- **CDN availability**: The SeQura logo is loaded from `live.sequracdn.com` at checkout time. If the CDN is unreachable, the `<img>` renders broken. Mitigation: add `alt="seQura"` so the text fallback is meaningful; no JS/PHP failure.
- **`props.components.PaymentMethodLabel` API stability**: This is an internal WooCommerce blocks component injected via props. If a future WC blocks update removes it, the Label falls back gracefully only if we guard with a null check. Add defensive fallback rendering bare `{label}` text.
- **`get_icon()` called before filter is evaluated**: In classic checkout, `get_icon()` is called when building the payment method list, which happens before the order exists. `is_delegated_payment_selection(null)` is safe (filter receives `null` order) — no regression risk.



- **Classic checkout hidden input auto-submit risk**: WooCommerce classic checkout calls `validate_fields()` server-side on submit. If the filter is toggled mid-session (page already loaded, then filter added), the hidden input won't be present in the DOM. This is acceptable since filter changes require a deployment, not a runtime toggle during a live session.
- **Block `canMakePayment` resolves `true` unconditionally in delegated mode**: If the filter is active but the seQura gateway is disabled in settings, the block still returns `true`. The `is_active()` check in WooCommerce blocks registration should prevent this, but the implementation should not bypass it. The existing `is_active()` method in `Sequra_Payment_Gateway_Block_Support` is called by WooCommerce before `canMakePayment` — confirmed safe.
- **`process_payment` skips `is_payment_method_data_valid` in delegated mode**: A malicious actor cannot exploit this to place orders with arbitrary product codes because they would still need a valid seQura solicitation and the seQura-hosted form enforces payment method selection. Risk is low.
- **`payment_fields()` in classic checkout still calls `try_to_get_order_from_context()`**: In delegated mode we still call this to get `$order` for the filter. The method is already safe to call returning `null`. No risk.
- **`delegatedPaymentMethodData` in `get_payment_method_data()` requires `$this->gateway` to be initialized**: `get_payment_method_data()` is called after `initialize()` which sets `$this->gateway`. Safe.
- **JS rebuild required**: Changing `payment-gateway.js` requires running `npm run build`. No automated step in the PHP test pipeline handles this; must be done separately before Playwright runs.

### Decisions

- `is_delegated_payment_selection()` lives on `Interface_Payment_Method_Service` so it can be mocked in PHPUnit; `Sequra_Payment_Gateway_Block_Support` calls `apply_filters` directly (consistent with existing `sequra_block_integration_is_solicitation_allowed` pattern there).
- Encoded `tbs` `Payment_Method_Data` is generated in PHP (`get_payment_method_data()`) and passed to JS as `settings.delegatedPaymentMethodData`, keeping the JS stateless about encoding.
- `validate_fields()` returns `true` early in delegated mode without making any solicitation call — avoids a redundant API request at validation time.
- `process_payment()` skips `is_payment_method_data_valid()` when delegated, but still calls `set_order_metadata()` with the tbs DTO — product `'tbs'` propagates naturally to order meta and then to `get_identification_form()` via `get_product()` → `wc_sequra_pumbaa_options` → integration-core URL.

## Testing Guidelines

- PHP quality checks with `./bin/phpcs`, `./bin/phpstan`, and `./bin/php-syntax-check --php=8.1`.
- PHPUnit coverage for any service or controller method that builds the checkout form URL or constructs the solicitation payload, verifying the `tbs` parameter appears when the filter is active and does not appear when inactive.
- Playwright E2E coverage: test the blocks checkout flow with the filter enabled (verify redirect to seQura form with `tbs` in URL) and with the filter disabled (verify existing behavior unchanged).
- Manual verification in the local Docker environment: enable the filter via `sequra-helper` or `functions.php`, proceed through blocks checkout, confirm the seQura form handles payment selection, and confirm the order completes successfully.
