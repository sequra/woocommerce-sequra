# Spec: Register StoreIntegration Via Migration

## Context / Background
The plugin configuration needs to be accessible through the seQura Merchant Portal. Currently, the StoreIntegration is not properly registered, preventing the Merchant Portal from accessing the plugin configuration. A migration is needed for version 4.3.0 to fix this.

## Summary
Implement a new migration for version 4.3.0 that ensures the StoreIntegration is properly registered so the plugin configuration is accessible through the Merchant Portal.

## Functional Requirements
- A new migration must register the StoreIntegration for the plugin
- The migration must run as part of the 4.3.0 version upgrade path
- After migration, the plugin configuration must be accessible through the Merchant Portal

## Non-Goals / Out of Scope
- Changes to the Merchant Portal itself
- Modifications to the StoreIntegration logic beyond registration

## Possible Edge Cases
- Existing installations that already have a correctly registered StoreIntegration
- Multisite installations where registration must apply per-store
- Migration running on a fresh install vs an upgrade

## Acceptance Criteria
- The migration executes successfully on upgrade to 4.3.0
- The StoreIntegration is registered after migration completes
- Plugin configuration is accessible through the Merchant Portal
- Existing correctly-registered installations are not disrupted

## Clarifications

- **Failure handling:** Non-critical — log error and continue. Plugin still works without Merchant Portal access.
- **No ConnectionData:** Return early silently — registration will happen naturally during onboarding.
- **Migration version:** `4.3.0` as stated in the ticket.

### 2026-04-22 — re-run clarifications

- **Skip existing:** Before calling `createStoreIntegration()`, check `StoreIntegrationRepositoryInterface::getStoreIntegration()`. If a record already exists, skip the API call entirely for that connection.
- **Loop resilience:** A failure on one connection must not abort the loop; catch the exception, log it, continue to the next connection.
- **Retry signal:** If any connection failed, throw a plain `Exception` (not `Critical_Migration_Exception`) after the loop completes. This causes `Migration_Manager` to skip `set_module_version('4.3.0')`, so the migration retries on the next plugin execution.

## Analysis

### Affected Files

**Bootstrap/registration:**

### Risks & Concerns

- **API failure during migration:** The `StoreIntegrationsProxy::createStoreIntegration()` call hits the seQura API. If the API is down or credentials are invalid, the migration will throw. Mitigation: catch the exception, log it, and continue (non-critical). After the loop, throw `Exception` so the migration retries on next plugin execution.
- **Idempotency:** `getStoreIntegration()` check before the API call ensures already-registered connections are not re-sent to the API. Safe to run multiple times.
- **Multisite:** Migrations run per-store via `StoreContext`. `StoreIntegrationRepositoryInterface::getStoreIntegration()` is scoped to the current store context, so the skip-existing check is per-store. _(2026-04-22: no change to the multisite risk; same mitigation applies.)_
- **Retry loop:** Throwing `Exception` prevents `set_module_version('4.3.0')` from being saved, so partially-registered stores will retry. Already-registered stores are skipped on retry (idempotent check). Risk is bounded.

### Decisions

- Follow Migration_Install_420 pattern: inject services via constructor, delegate to core service layer.
- Use `ConnectionService::getAllConnectionData()` to retrieve connections (new format, post-4.0.0).
- Call `StoreIntegrationService::createStoreIntegration()` for each ConnectionData.
- Wrap in try/catch per connection so one failure doesn't block others.
- _(2026-04-22)_ Inject `StoreIntegrationRepositoryInterface` directly (available via `Reg::getService`) to perform the skip-existing check before calling the service. The service already does this check internally but would still hit the API; we skip entirely at the migration level.
- _(2026-04-22)_ Use a `$has_failures` flag: set to `true` on any caught exception, throw `Exception` (not `Critical_Migration_Exception`) after the loop if `$has_failures` is `true`.
